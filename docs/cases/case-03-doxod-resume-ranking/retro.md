# Сложности и рефлексия

## Сложности и как справились

### Проблема 1: Ограничения hh.ru API

- **В чём была:** у employer-API hh.ru есть rate limits, квоты, а часть полей доступна только через отдельные эндпоинты (сначала получи список откликов, потом по каждому отклику — полное резюме). При большом потоке откликов наивный «в лоб» пулл быстро упирался в лимиты, а часть резюме возвращалась неполной.
- **Как обнаружили:** на первых нагрузочных прогонах на реальных вакансиях — начали получать 429 и неполные тела резюме.
- **Что сделали:** на стороне бэкенда сделали **кэширование** уже вытянутых резюме, **экспоненциальный бэкофф и ретраи**, аккуратную **склейку полей** из нескольких эндпоинтов. Пулл был распределён во времени: не «весь список разом», а по мере необходимости под конкретную вакансию.
- **Результат:** сервис перестал упираться в лимиты, данные стабильно доходят до ранжировщика в полном виде.

### Проблема 2: Latency LLM-reranker'а (Qwen 72B)

- **В чём была:** Qwen 72B — большая модель, даже на SGLang один вызов по паре «вакансия ↔ резюме» занимает заметное время. Если прогонять через неё всех кандидатов, которых нашли BM25 и Qdrant (десятки–сотни на вакансию), HR ждал бы сортировку минутами, а SLA на отклик из UI — секунды.
- **Как обнаружили:** на первых end-to-end прогонах ранжирования одной вакансии — суммарное время резко росло с увеличением числа кандидатов на вход reranker'а.
- **Что сделали:**
    - Жёстко ограничили **top-K = 30–50** кандидатов, идущих в reranker (после гибридной кандидатогенерации). Это позволяет отсечь заведомо нерелевантные резюме дешёвыми BM25/dense, а дорогую LLM пускать только на реально конкурирующий топ.
    - **Батчинг** запросов через SGLang — несколько пар «вакансия ↔ резюме» идут в одну пачку, что даёт значимый выигрыш по throughput.
    - Оптимизировали промпт — убрали лишний контекст, оставили только ключевые поля резюме и структурированного профиля вакансии.
- **Результат:** ранжирование одной вакансии стало укладываться в приемлемый для UI таймаут; качество верхних позиций при этом не пострадало.

### Проблема 3: Мало размеченных данных для дообучения эмбеддера

- **В чём была:** исторических наймов в компании — ограниченное число, и не все из них «чистые» позитивы (кого-то нанимали «за отсутствием других», кого-то уводили с соседней команды). Для contrastive-лосса нужны качественные пары и адекватные негативы, а их банально мало.
- **Как обнаружили:** при первом заходе на дообучение — модель переобучалась и плохо обобщалась на новых вакансиях.
- **Что сделали:**
    - Сгенерировали **синтетические пары вакансия↔резюме** через self-hosted LLM: LLM по описанию вакансии генерировала «сильные» и «слабые» резюме, которые потом попадали в обучение с соответствующими лейблами.
    - Подняли **hard-negative mining**: для каждой пары брали резюме, которые BM25/базовый dense считали похожими, но по явным полям (грейд, стек, домен) они не подходили. Это заставило эмбеддер учиться **различать похожие, но не совпадающие по смыслу** резюме.
- **Результат:** эмбеддер стал устойчивее на новых вакансиях, NDCG@k на hold-out'е заметно подрос по сравнению с базовой моделью «из коробки».

## Что бы сделал иначе

- **Раньше зафиксировал бы формат структурированного профиля вакансии.** Часть первых итераций ушла в то, что HR заполнял профиль по-разному, и одни и те же требования писались разными словами. Жёсткий шаблон с самого начала сэкономил бы время.
- **Начал бы с простейшего baseline** — только дообученный dense + RRF без LLM-reranker'а — и смотрел, где именно ломается качество. Возможно, удалось бы обойтись меньшей моделью для reranker'а в части вакансий.
- **Собирал бы HR-фидбэк через UI изначально**, а не постфактум. Каждое решение HR «подходит / не подходит» — это потенциальная обучающая пара, и их стоило начать копить с первого дня.
- **Отдельный бенчмарк на синтетике vs реальных наймах** — чтобы честно понимать, какая доля качества пришла из реальных данных, а какая из LLM-синтетики.

## Чему научился лично

- **Гибридный IR на практике.** Не только «запустил BM25 и embeddings», но и понимание, когда что выигрывает, и почему RRF — хорошее значение по умолчанию.
- **LLM как reranker, а не как генератор.** Отдельный скилл — формулировать промпт для пары, фиксировать формат ответа, подбирать размер модели под стабильность скоров.
- **SGLang и инфраструктура инференса.** Батчинг, структурированная генерация, ограничения по памяти GPU при 72B.
- **Работа с HR как со стейкхолдером.** Как переводить требования HR («хочу, чтобы сверху были сильные») в формальные метрики (NDCG@k) и обратно.
