# Технологический стек

Всё self-hosted в контуре компании, весь стек — opensource.

## Основной стек

| Инструмент | Лицензия | Назначение | Почему именно он |
|---|---|---|---|
| Python 3.x | PSF | Основной язык | Стандарт для ML/NLP и интеграций |
| **hh.ru employer API** | — | Источник вакансий, откликов и резюме | Официальный канал — единственный легальный способ получать данные откликов |
| **rank_bm25** | Apache 2.0 | Лексическая кандидатогенерация (BM25) | Чистый Python, без внешних сервисов, достаточно для in-process индекса на масштабе нашего корпуса резюме |
| **sentence-transformers** | Apache 2.0 | Базовый фреймворк эмбеддера | Стандарт для contrastive-дообучения, совместим с русскоязычными моделями |
| Дообученный sentence-transformer | Apache 2.0 (веса базовой модели) | Эмбеддер резюме и вакансий | Базовая модель не знала специфики HR-домена и русского финтеха → дообучили на исторических наймах + синтетике |
| **Qdrant** | Apache 2.0 | Векторная БД, ANN-поиск по dense-эмбеддингам | Opensource, self-hosted, хорошо ложится на on-prem, простой API, payload-фильтры |
| **SGLang** | Apache 2.0 | Инференс-рантайм для LLM | Высокая пропускная способность, батчинг, структурированная генерация — идеально для reranker-сценария |
| **Qwen 72B** (open weights) | Tongyi Qianwen License | LLM-reranker | Крупная модель даёт стабильные и объяснимые скоры на парах вакансия↔резюме; 7B/14B/32B давали шумное ранжирование |
| Self-hosted LLM | — | Генерация синтетики для дообучения эмбеддера | Нельзя слать резюме и истории наймов во внешние API — только локально |
| **FastAPI** | MIT | REST-сервис ранжировщика | Быстро, async, OpenAPI для frontend'а |
| Docker | Apache 2.0 | Упаковка сервисов | Стандарт для on-prem деплоя |

## Почему такая архитектура (а не что-то проще)

- **Только BM25** — ловит буквальные совпадения, но проигрывает на синонимах и смысле («Python-разработчик» vs «бэкенд на Питоне»). HR жаловались бы на пропуски.
- **Только dense-эмбеддинги** — наоборот, иногда недооценивают буквальные совпадения редких навыков (специфичный фреймворк, название системы). BM25 вытягивает такие кейсы.
- **Гибрид BM25 + dense** даёт устойчивую кандидатогенерацию, но обе ветки всё ещё шумят на ранжировании верхних позиций.
- **LLM-reranker сверху** нужен именно для верхних 30–50 кандидатов: там важна точность сортировки, и большая модель делает это хорошо + даёт объяснение для HR.

## Почему Qwen 72B, а не 7B/14B/32B

Меньшие версии Qwen на тех же промптах давали **шумные и нестабильные скоры**: на слегка перефразированной вакансии ранжирование верхних кандидатов заметно менялось, а объяснения были формальными и часто повторяли текст резюме вместо анализа. 72B даёт заметно более стабильное ранжирование и связные объяснения. Цена — latency, но она амортизируется ограничением top-K (30–50) и батчингом в SGLang.

## Почему НЕ внешние LLM-API

- **ПДн** — резюме содержат персональные данные кандидатов, слать их во внешние облака нельзя.
- **Внутренняя политика** — on-prem без выхода к публичным LLM.
- Self-hosted Qwen + SGLang решает это полностью.

## Почему НЕ Elasticsearch/OpenSearch

Для in-process BM25 нашего масштаба хватает `rank_bm25` — поднимать и поддерживать отдельный поисковый кластер было бы оверкиллом. Dense-часть уехала в Qdrant, где и так есть всё нужное для ANN.
