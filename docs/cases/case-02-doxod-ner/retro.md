# Сложности и рефлексия

## Сложности и как справились

### Проблема 1: Курица-и-яйцо — нет размеченных данных, и они не могут появиться

- **В чём была:** чтобы обучить NER, нужна размеченная выборка. Реальные документы содержат ПДн, и их нельзя выносить разметчикам ни внутрь Label Studio с внешним доступом, ни в облако, ни даже просто «на соседний компьютер» — комплаенс запрещает. Получался замкнутый круг: чтобы убрать ПДн из документов, нужна модель; чтобы обучить модель, нужны документы с ПДн.
- **Как обнаружили:** сразу, на этапе планирования проекта — когда обсуждали, как будем размечать.
- **Что сделали:** пошли через **синтетику**. Отобрали 10–20 шаблонов реальных документов, через **self-hosted LLM** (внутри контура) генерировали вариативные тексты по этим шаблонам, а ПДн-значения (ФИО, ИНН, паспорта, номера счетов, суммы) подставляли алгоритмически — так ground-truth метки были точно известны и не требовали ручной разметки. Добавили открытые русскоязычные NER-датасеты для устойчивости. Итого ~10–30 тыс. обучающих примеров без единого реального клиента в датасете.
- **Результат:** получили обучающую выборку нужного объёма без нарушения комплаенса. На hold-out'е из внутренних документов (небольшой, размеченный руками внутри контура) модель показала Recall 0.90–0.95.

### Проблема 2: On-prem без доступа в интернет

- **В чём была:** стандартные рабочие процессы DS-инженера ломаются: нельзя `pip install` из PyPI, нельзя `huggingface_hub.download`, нельзя зайти в документацию, нельзя подтянуть новые веса. Любая зависимость и любой артефакт должны быть заранее подготовлены и перенесены в контур.
- **Как обнаружили:** с первого дня, это было изначальное ограничение компании.
- **Что сделали:** собрали полный набор artefact'ов в подготовительной среде с интернетом — wheel'ы всех python-зависимостей, deb-пакеты Tesseract, веса базового GliNER, открытые NER-датасеты — и перенесли в контур как единый артефакт. Обновления стали дискретными событиями, а не «поставил и забыл».
- **Результат:** проект запустился и поддерживается внутри контура без необходимости внешних запросов.

### Проблема 3: Нельзя использовать PyMuPDF (AGPL)

- **В чём была:** PyMuPDF — очевидный первый выбор для PDF-редактирования, у него есть готовый redaction API. Но лицензия AGPL-3.0 несовместима с коммерческим проприетарным использованием без покупки лицензии у Artifex.
- **Как обнаружили:** на этапе выбора стека, при согласовании с юристами/безопасностью.
- **Что сделали:** собрали собственный redaction-пайплайн из BSD/MIT-компонент: **pdfplumber** (извлечение слов с bbox) → **reportlab** (генерация overlay-страницы с чёрными прямоугольниками) → **pypdf** (merge overlay с исходником). Для критичных документов предусмотрели дополнительный шаг растеризации страниц, чтобы текстовый слой гарантированно исчезал.
- **Результат:** лицензионно чистый пайплайн, работает единообразно для текстовых и отсканированных PDF.

### Проблема 4: Качество OCR на плохих сканах

- **В чём была:** для сканов входной текст получался из Tesseract. На плохих сканах (перекос, низкое разрешение, шум, печати поверх текста) OCR давал ошибки и в самом тексте, и в bbox. NER по «битому» тексту пропускал часть сущностей — а это прямой удар по recall, главной метрике.
- **Как обнаружили:** при валидации на внутренних документах — на сканах recall заметно проседал по сравнению с текстовыми PDF.
- **Что сделали:** добавили **препроцессинг изображений перед OCR** через Pillow (deskew, бинаризация, удаление шума). Для неуверенных страниц — пересборка с другими параметрами Tesseract. Для особо плохих сканов — флаг «проверить вручную», чтобы не тихо пропускать ПДн.
- **Результат:** recall на сканах подтянули ближе к уровню текстовых PDF, а плохие страницы перестали молча пропускать сущности — они явно помечаются для ревью.

## Что бы сделал иначе

- **Начал бы сразу с синтетики** и не тратил время на попытки «как-то договориться о разметке реальных документов» — это был тупик с самого начала.
- **Раньше зафиксировал бы requirements по лицензиям** со стороны безопасности и юристов — тогда PyMuPDF вообще не попал бы в план как кандидат, и не пришлось бы переделывать PDF-часть.
- **Собрал бы отдельный бенчмарк по качеству OCR** до того, как строить пайплайн — чтобы сразу видеть, где просадка от OCR, а где от NER, и не искать причину в модели.
- **Договорился бы о политике явных флагов** («не уверен — пометить на ревью») раньше, а не в ответ на найденные проблемы.

## Чему научился лично

- **Технически:** дообучение GliNER на русскоязычных данных, генерация реалистичной синтетики через LLM с контролируемыми ground-truth метками, ручная сборка PDF-redaction-пайплайна из примитивов (без PyMuPDF).
- **Инфраструктурно:** работа в on-prem без интернета — офлайн-поставка зависимостей, весов, датасетов.
- **Продуктово:** важность выбора правильной главной метрики (recall, а не F1) и умение объяснить этот выбор бизнесу — «пропуск ПДн дороже, чем лишняя замазка».
- **Юридически:** научился читать лицензии opensource-библиотек и различать, что можно тащить в коммерческий продукт, а что нет.
