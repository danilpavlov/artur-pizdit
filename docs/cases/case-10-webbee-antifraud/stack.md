# Технологический стек

| Инструмент | Назначение | Почему именно он |
|---|---|---|
| Python 3.x | Основной язык | Стандарт ML и сервисов |
| **CatBoost / LightGBM** | Supervised-модель риска транзакции | Быстрая, хорошая работа на табличных данных с категориальными (BIN, страна, device-тип), устойчивы к дисбалансу через `scale_pos_weight`, нативно дают feature importances |
| scikit-learn | **Isolation Forest**, препроцессинг, метрики | Стандартное API; Isolation Forest — хороший дефолт для unsupervised-детекции аномалий на табличных данных |
| (опц.) PyTorch + autoencoder | Альтернатива Isolation Forest для новых паттернов | Если табличный autoencoder показывает лучшие результаты на специфике клиента — используется в связке с IF |
| **networkx** (или graph-tool на больших объёмах) | Построение и работа с графом сущностей | Чистый Python, удобно для ночного batch'а; graph-tool — если узлов станет больше миллиона |
| **python-louvain / community** | Community Detection алгоритмом Louvain | Быстрый, проверенный, хорошо работает на реалистичных графах мошеннических колец; даёт стабильные сообщества по модулярности |
| **SHAP** | Объяснение решений модели | Локальные объяснения по транзакции → текстовые причины для антифрод-аналитика и для споров с банком |
| **Redis** | Быстрый key-value кэш предвычисленных графовых агрегатов | Ключевой элемент latency-бюджета: без Redis графовые фичи не успеют считаться inline в чекауте |
| **FastAPI** | Realtime-сервис скоринга | Async, OpenAPI, быстрый response time, удобная интеграция с бэком клиента |
| Uvicorn / Gunicorn | ASGI-сервер | Стандарт для FastAPI в продакшене |
| PostgreSQL / DWH | История транзакций, клиенты, фичи | Основной источник офлайн-данных для retrain и batch-графа |
| **Airflow** | Оркестрация ночного batch (граф, community, retrain, обновление Redis) | Уже был в инфре Webbee, хорошо ложится на batch-сценарии |
| **MLflow** | Трекинг экспериментов + Model Registry | Версионирование моделей, воспроизводимость, realtime-сервис читает текущую версию из registry |
| **Evidently** | Мониторинг drift'а фичей и скоров | Opensource, интеграция с Airflow, алерты при значимом сдвиге |
| Grafana / Prometheus | Мониторинг production-метрик (latency, RPS, доля allow/3DS/block) | Стандартная наблюдаемость |
| Docker | Упаковка сервиса | Стандарт |

## Почему такая архитектура (а не одна модель)

**Альтернатива 1 — только supervised CatBoost.**
Плохо потому что:
- Ловит только **известные паттерны** фрода из истории chargeback'ов; новые схемы пропускает.
- Индивидуальная транзакция может выглядеть чисто, а графовый контекст — нет.
- Разметка приходит с задержкой 30–90 дней (chargeback cycle), поэтому модель всегда немного «отстаёт».

**Альтернатива 2 — только правила.**
Плохо потому что:
- Не масштабируется: каждая новая схема → новое правило; правила начинают конфликтовать.
- Не ловит комбинации слабых сигналов.
- Не даёт осмысленной вероятности фрода — только бинарное срабатывание.

**Альтернатива 3 — только графовый анализ.**
Плохо потому что:
- Хорошо ловит организованный фрод (кольца), но **пропускает одиночных мошенников** (первый раз в системе, чистый граф).
- Строить community realtime нельзя — только офлайн.

**Выбранная многослойная архитектура:**
- **Supervised** — основа; закрывает большую часть известного фрода.
- **Unsupervised** — ловит новое; страховка от устаревания supervised-модели.
- **Графовый** — даёт уникальные сигналы про связанный фрод, которые никак иначе не получить.
- **Правила поверх** — белые/чёрные списки, жёсткие override'ы, бизнес-ограничения.
- **Агрегатор** — учится правильно их комбинировать.

Каждый слой можно валидировать и отлаживать отдельно, каждый имеет свою метрику и свою роль.

## Почему Redis, а не считать граф в realtime

Community Detection — алгоритм **глобальный**: ему нужен весь граф, а не только один узел. Запускать Louvain на каждый запрос — это секунды-минуты, что никак не укладывается в inline SLA чекаута (десятки-сотни миллисекунд).

Решение: **граф считаем ночью**, агрегированные метрики по каждому узлу (community_id, размер community, доля фрода в community) кладём в Redis с ключами по узлу, realtime-сервис **только читает** эти готовые значения за миллисекунды.

Актуальность графовых фичей — сутки; этого для фрод-сценария достаточно, потому что сама структура community меняется медленно.

## Почему Isolation Forest, а не ещё один бустинг

Нужен **unsupervised**-сигнал — модель, которая **не знает** заранее, что такое фрод, и флагает просто «аномальное» в распределении фичей. Isolation Forest хорош для этой роли:

- быстрый в инференсе (вписывается в realtime-бюджет);
- не требует разметки;
- стабильно ловит «всё необычное» — даже то, что никогда не было подтверждённым chargeback'ом.

Альтернативы (Autoencoder, One-Class SVM) — сложнее в эксплуатации и без явного выигрыша по качеству на табличных данных такого масштаба.

## Что НЕ использовали и почему

- **Deep learning на транзакциях.** Избыточно для табличной задачи, медленнее в realtime, хуже интерпретируется — бустинг закрыл задачу.
- **Стриминговые Graph DB** (Neo4j + live графовые запросы в realtime). Дорого и сложно в эксплуатации; наша задача прекрасно решается офлайн-графом + Redis.
- **Only rules-based движок.** Простое правило вроде «> 3 попыток за минуту» даёт 20–30% эффекта, но не закрывает основную массу современного фрода.
