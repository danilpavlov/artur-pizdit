# Контекст

## Webbee ML и клиент

**Webbee ML** — аутсорс ML для e-commerce. Проектная модель работы: под каждого клиента своя небольшая команда.

**Клиент** — **D2C-магазин**, торгующий через собственный интернет-магазин (не через маркетплейс). Свой чекаут, собственный платёжный процессинг через внешнего эквайера. Название и категория обезличены по NDA. Масштаб — сотни/тысячи транзакций в день, значимая доля зарубежных карт.

## Бизнес-задача

До проекта у клиента был классический набор «платёжных болей»:

- **Чарджбэки.** Регулярные оспаривания транзакций — часть от реального фрода (украденные карты, account takeover), часть «friendly fraud» (клиент получил товар и оспорил платёж).
- **Прямые убытки.** Каждый chargeback — это не только возвращённая сумма, но и комиссия эквайера + репутационная метка.
- **Риск попасть в программу повышенного риска** у платёжных систем (Visa / Mastercard): при высокой доле чарджбэков эквайер поднимает комиссию или вовсе отказывается обслуживать магазин.
- **Ручные правила** не справлялись: клиент крутил простые эвристики («не больше N заказов с одной карты в час», «блокировка по стране»), но они либо пропускали реальный фрод, либо ложно блокировали честных покупателей.
- **Перегруженный антифрод-аналитик** клиента — один человек, который вручную разбирал подозрительные транзакции и договаривался с эквайером по каждому спору.

Задача — **автоматическая realtime-система**, которая на этапе чекаута:

1. Оценивает риск транзакции и выдаёт решение **`allow / 3DS / block`**.
2. **Не тормозит** честных клиентов (жёсткое ограничение FPR).
3. **Ловит не только известные паттерны**, но и новые схемы фрода.
4. **Даёт аналитику объяснение**, чтобы он мог разбирать спорные кейсы и спорить с банком.

## Данные

- **История транзакций:** все платежи за последние годы с суммами, валютами, способами оплаты, статусами (success/fail/chargeback) и `reason_code` от эквайера.
- **Клиенты и аккаунты:** ID, email, телефон, возраст аккаунта, история заказов, device fingerprint'ы, IP, user-agent.
- **Сессионные данные с фронта:** поведение клиента на сайте до оплаты — клики, скорость ввода, автозаполнение vs ручной ввод, ошибки форм, источник трафика, referer.
- **Платёжные метаданные от эквайера:** BIN карты, страна эмитента, результат 3DS, предыдущие попытки по этой карте.
- **История chargeback'ов** — главный источник разметки; приходит с задержкой **30–90 дней** (dispute cycle банка).
- **Ручные вердикты антифрод-аналитика** — дополнительный источник разметки для сомнительных транзакций и «свежих» случаев, где chargeback ещё не дошёл.

## Почему ML, а не только правила

- **Правила не масштабируются.** Каждая новая схема фрода требует новой эвристики, и они конфликтуют.
- **Правила ловят только буквальные совпадения.** Модель умеет комбинировать сотни слабых сигналов в один скор, и такие комбинации правилами не покрыть.
- **Правила не учитывают графовый контекст.** Индивидуальный клиент может выглядеть чисто, а его устройство/карта/email — фигурировать в сети, где уже были подтверждённые чарджбэки. Это ключевой сигнал фрода, и правилами такое почти невозможно поддерживать.
- **Unsupervised-слой** ловит новое — то, для чего ещё нет разметки и никогда не будет правила.

## Ограничения

- **Realtime inline SLA.** Сервис вызывается синхронно на этапе чекаута. Если ML тормозит, чекаут тормозит, бизнес теряет выручку. Десятки-сотни миллисекунд на p99.
- **Жёсткий FPR-контроль.** Ложно заблокированный честный клиент — это упущенная выручка плюс репутационный удар. FPR ограничен сверху согласованным с клиентом порогом.
- **Chargeback-задержка 30–90 дней.** Разметка «это был фрод» приходит с лагом. Нужна поддержка актуальности модели без ожидания полного цикла.
- **Explainability.** Антифрод-аналитик клиента должен понимать каждое решение модели — и для внутренней работы, и для споров с банком.
- **Обезличенность.** Клиент и категория не раскрываются.

## Моя роль (Артур)

**Единственный Data Scientist на проекте.** Отвечал за всю ML-часть:

- supervised CatBoost/LightGBM-модель на фичах транзакции, клиента, устройства и графа;
- unsupervised-слой (Isolation Forest / Autoencoder) для новых паттернов;
- графовый пайплайн: построение графа, Community Detection, агрегированные фичи по сообществам;
- фичи, валидация, тюнинг порога под целевой FPR, метрики (PR-AUC + бизнес-эффект в $);
- SHAP-объяснения и текстовые причины для UI антифрод-аналитика;
- постоянный retrain и мониторинг drift'а.
