# Технологический стек

| Инструмент | Назначение | Почему именно он |
|---|---|---|
| Python 3.x | Основной язык | Стандарт ML/DS |
| pandas / numpy | Обработка табличных данных | Базовый стек для feature engineering |
| scikit-learn | Препроцессинг, метрики, пайплайны | Единое API, совместимо с бустингами |
| **hdbscan** | Кластеризация клиентов | HDBSCAN даёт сегменты несферической формы разной плотности + явно выделяет «шумовых» клиентов (новички/одноразовые), которых нельзя насильно загонять в кластеры |
| UMAP (опц.) | Снижение размерности перед HDBSCAN | Устойчивее, чем PCA, для кластерных структур |
| **CatBoost** / **LightGBM** | Скоринг вероятности отклика на каждый продукт | Быстрые, хорошо работают на табличных данных, нативно обрабатывают категориальные признаки, устойчивы к несбалансированным классам через `class_weight`/`scale_pos_weight` |
| **SHAP** | Интерпретация моделей для маркетинга | Позволяет объяснить и глобальную важность (портреты сегментов), и локальные причины скоров конкретных клиентов |
| **MLflow** | Трекинг экспериментов и Model Registry | Воспроизводимость, версионирование моделей, хранение метрик по каждому продукту отдельно |
| **Airflow** | Оркестрация ночного batch-DAG'а | Уже был в инфраструктуре компании, хорошо ложится на batch-сценарий |
| Feature store | Витрина фичей по состоянию на дату T | Критичен для time-aware split'а и устранения target leakage — фичи на обучение и инференс должны браться «на момент» |
| Evidently / Grafana | Мониторинг drift'а и бизнес-метрик | OSS, легко интегрируется с Airflow |
| Триггеры realtime (Kafka / CDC / webhook) | Доставка событий клиента в realtime-ветку | Ключевые события (пополнение, первая покупка, смена риск-профиля) должны обрабатываться в минутах, не ждать ночного batch'а |
| PostgreSQL / DWH | Хранение витрин NBA и feature store | Корпоративный стандарт |

## Почему такая архитектура

- **Почему не один мультиклассовый классификатор** на «какой из продуктов купит»? Во-первых, клиент реально может купить сразу несколько, и multi-class это плохо моделирует. Во-вторых, отдельные модели на продукт дают лучший контроль над метриками каждого оффера и возможность дообучать независимо.
- **Почему HDBSCAN + супервайзд, а не только супервайзд?** Сегментация даёт маркетингу **объяснение и портрет клиента**, которого просто скоры дать не могут. Маркетологу легче работать с сегментами «молодые активные спекулянты» и «консервативные держатели ПИФ», чем с абстрактными вероятностями.
- **Почему CatBoost/LightGBM, а не нейросеть?** Данные табличные, объёмы средние, бустинги стабильнее, быстрее учатся и проще интерпретируются через SHAP. Нейросеть здесь — оверкилл.
- **Почему batch + realtime, а не только одно?** Только batch — CRM реагирует с задержкой в сутки и пропускает «горячие» моменты. Только realtime — дорого и избыточно, большинство клиентов не меняют поведение каждый день. Гибрид даёт и полноту, и скорость реакции.
