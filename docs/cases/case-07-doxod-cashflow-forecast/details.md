# Детали

## Технологический стек

| Инструмент | Назначение | Почему |
|---|---|---|
| Python 3.x | Основной язык | Стандарт DS |
| pandas / numpy | Работа с рядами, лаги, окна | Стандарт для подготовки временных рядов |
| **Prophet** | Baseline-модель с сезонностью и holidays | Быстрый, надёжный, отлично ловит недельную/годовую сезонность и кастомные праздники |
| **statsmodels (ETS)** | Ещё один baseline | Классика для sanity check |
| **LightGBM** | Основная ML-модель на лаг-фичах | Быстрая, хорошо работает с экзогенными фичами, нелинейные взаимодействия |
| scikit-learn | Метрики, пайплайны, `TimeSeriesSplit` | Единое API |
| **Airflow** | Оркестрация ежедневного переобучения и прогноза | Уже был в инфре |
| Grafana / Superset | Дашборд казначейства | Стандарт визуализации |
| Парк праздников / календарь РФ | Источник календарных фичей | Нужно корректно ловить перенесённые выходные и короткие дни |

## Команда и роль Артура

- **Data Scientist (Артур)** — постановка задачи с казначейством, feature engineering, построение и сравнение моделей (Prophet / ETS / LightGBM), rolling-origin бэктест, интеграция с Airflow и дашбордом.
- **Data Engineer** — источник исторических пополнений/выводов, календарные справочники.
- **Казначейство / фин. отдел** — заказчик; формулирует требования к точности и горизонту, принимает результат.
- **Маркетинг** — потребитель прогноза для планирования коммуникаций.

## Сложности

1. **Календарные аномалии.** Стандартные праздничные календари Prophet'а не учитывают специфику РФ (перенос выходных, короткие предпраздничные дни, налоговые даты). Пришлось собирать **собственный календарь** и подавать как holidays в Prophet + как флаги в LightGBM.
2. **Рыночные шоки.** На периодах резких движений (регуляторные новости, кризисы) модель может дать бессмысленный прогноз. Решение — **индикаторы волатильности** как экзогенная фича + **доверительные интервалы**, которые расширяются в волатильные периоды (дашборд явно показывает неопределённость).
3. **Разные горизонты — разные модели.** Попытка сделать одну универсальную модель на все горизонты давала посредственное качество везде. В итоге сделали **разные модели под D+1..D+14 и W+1..W+8**, с разными наборами фичей и разной частотой переобучения.
4. **Target leakage на лагах.** При recursive-forecasting легко забыть, что «лаг t−1» на прогнозе t+5 — это уже предсказание, а не факт. Покрыли это явной проверкой в бэктесте: сравнивали recursive vs direct, убеждались, что recursive не «жульничает».

## Что бы сделал иначе

- С самого начала **отдельные модели на приток и отток** (а не одна на «нетто»): поведение разное, сезонности разные, смешивать их в одну модель — терять сигнал.
- Раньше настроил бы **алерты на большие ошибки прогноза** — чтобы казначейство знало, когда модели не стоит доверять (волатильный период, нетипичные события).
- Завёл бы **дашборд качества прогноза** в продакшне (plot «факт vs прогноз» с ошибками по дням) с первого дня, а не после запроса от казначейства.
