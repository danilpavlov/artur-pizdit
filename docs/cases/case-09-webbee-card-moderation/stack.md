# Технологический стек

| Инструмент | Назначение | Почему именно он |
|---|---|---|
| Python 3.x | Основной язык | Стандарт ML/CV/NLP |
| **YOLO-family** (YOLOv8/YOLOv10) | Детектор чужих логотипов и водяных знаков | Индустриальный стандарт object detection, быстрый инференс, удобное дообучение на своих данных, большая экосистема |
| **OpenNSFW / NudeNet** | NSFW-классификатор фото | Готовые opensource-веса, закрывают задачу с минимальной настройкой — тюнятся только пороги под recall |
| Небольшой CNN + OpenCV | Проверка качества фото (блюр, фон, ориентация) | Вариация Лапласиана для блюра + лёгкий CNN для остального — быстро, надёжно, не нужно большой модели |
| **CLIP-подобная модель** (русскоязычная мультимодалка / image+text encoder) | Проверка соответствия фото описанию | Мультимодальные эмбеддинги — индустриальный дефолт для cross-modal similarity, поддерживают cosine similarity без специального train loop |
| Доменный fine-tune CLIP-модели | Уточнение под категории клиента | На стандартных весах мультимодалка универсальна, но плохо ловит тонкие категории e-com; fine-tune на истории одобренных карточек закрыл это |
| **sentence-transformers** / классификатор на BERT | NLP-проверка описаний (спам, контакты, соответствие атрибутам) | Стандарт для русскоязычной классификации текста, хорошо дообучается на разметке модераторов |
| Regex + накопленные словари | Быстрый слой на очевидные обходы | Ловит классические обходы (телефоны, ссылки) за миллисекунды без запуска моделей; +расширяется auto-mining'ом |
| **LightGBM** | Агрегатор вердиктов | Табличная задача на скорах и метаданных, нативные категориальные, быстрый, интерпретируемый — идеально для «объединяющей» модели поверх детекторов |
| **FastAPI** | REST-сервис модерации | Async, OpenAPI, удобен для параллельного запуска голов внутри запроса |
| Docker | Упаковка и деплой | Стандарт |
| PyTorch | Рантайм CV-моделей | Base-фреймворк для YOLO, CLIP-like и CNN |
| **Label Studio** | Разметка и чистка данных | Opensource, self-hosted — можно развернуть в контуре клиента, удобен для многоклассовой разметки фото и текста |
| Airflow | Регулярный retrain и обновление словарей | Уже был в инфре Webbee |
| Grafana / dashboards | Мониторинг качества модели и production-метрик | Стандарт наблюдаемости |

## Почему многоголовая архитектура, а не одна multi-modal модель

**Альтернатива 1 — один большой multi-modal классификатор** (CLIP + text → один бустинг «approve/reject»).
Плохо потому что:
- Теряется **объяснимость**: модератору нужны конкретные причины, а не просто «плохо».
- Нельзя настроить **разный recall-precision trade-off** на разные типы проверок (на запрещёнке нужен жёсткий recall, на качестве текста можно мягче).
- Любая ошибка одной проверки ломает всю модель, тогда как в многоголовой её можно точечно дообучить или перетюнить.

**Альтернатива 2 — каскад «если X то Y»** без ML-агрегатора.
Плохо потому что:
- Разные проверки разной надёжности — жёсткая логика «OR» даёт слишком много `reject`.
- Метаданные (категория, поставщик) сильно влияют на интерпретацию — в правилах это выливается в сотни exception'ов.

**Выбранная многоголовая архитектура с LightGBM-агрегатором:**
- Каждая голова закрывает свою задачу и отлаживается отдельно.
- Агрегатор учится правильно взвешивать сигналы на реальной разметке модераторов.
- Объяснение вердикта формируется естественно — по флагам отдельных голов.
- Можно по очереди улучшать компоненты: обновили CV-детектор логотипов — переобучили только агрегатор, остальное не трогали.

## Почему YOLO, а не классификатор

**Классификатор «есть логотип / нет»** работает, но не даёт **локализации** (bbox) — модератору непонятно, на каком из 5 фото карточки и где именно проблема. **YOLO-детектор** возвращает bbox с координатами, что даёт готовое объяснение в UI: «посмотри на фото 2, красная рамка — это и есть найденный чужой логотип».

## Почему LightGBM как агрегатор, а не нейросеть

- Агрегатор работает с **десятками табличных фичей** (скоры голов + метаданные) — нейросеть тут избыточна.
- LightGBM **быстрее** (важно для inline pre-publish), **интерпретируемее** (SHAP по фичам), **проще обучать** на небольшом датасете разметки модераторов.
- Модель легко пересобирать при изменении набора голов — без переобучения тяжёлых CV/NLP моделей.

## Что НЕ использовали и почему

- **LLM для модерации текста.** Это выглядит красиво, но дорого по latency/inference на потоке тысяч карточек и избыточно для задач классификации «спам / контакт / нормально» — sentence-transformer закрывает их кратно дешевле и детерминированнее.
- **OCR для поиска текста на фото.** Обсуждался для детекции номеров на картинках — в итоге заменён связкой YOLO-детектор watermark-зон + NLP-постфильтр на том, что нашёл OCR только на подозрительных карточках (каскад).
- **Чистый rule-based движок.** Покрывает только 20–30% реальных кейсов, всё остальное требует ML.
