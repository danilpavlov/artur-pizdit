# Детали

## Технологический стек

| Инструмент | Назначение | Почему |
|---|---|---|
| **PySpark** | Основной фреймворк обработки | Масштабирование на миллионы транзакций, в pandas не влезает |
| Кластер Spark (YARN / Kubernetes) | Рантайм | Уже была инфра в компании |
| **Parquet** | Формат выходной витрины | Колоночный, быстро читается, нативно поддерживается Spark/pandas/ML-фреймворками |
| Hadoop/S3-совместимое хранилище | Хранение Parquet | Стандарт для big-data пайплайнов |
| **Airflow** | Оркестрация ночного DAG'а | Уже был в инфраструктуре, хорошо ложится на batch |
| `SparkSubmitOperator` | Запуск джобы из Airflow | Родной оператор для Spark |
| **pytest + chispa** (или `pyspark.testing`) | Юнит-тесты PySpark-преобразований | Небольшие sample-датафреймы, быстрые проверки на CI |
| JDBC-коннектор к DWH | Чтение транзакций | Единственный доступный канал до части источников |

## Команда и роль Артура

- **Data Scientist / Data Engineer (Артур)** — проектирование схемы витрины, реализация PySpark-джобы, оптимизация джоинов и партиционирования, юнит-тесты, интеграция с Airflow.
- **Data Engineer команды** — доступ к источникам, поддержка DWH, настройка кластера.
- **Команды ML-кейсов (NBA, churn)** — потребители витрины, заказчики новых фичей.

## Сложности

1. **Skew на VIP-клиентах.** У части клиентов сотни тысяч транзакций — при джоине на `client_id` их партиции становились в разы тяжелее остальных, джоба зависала на последнем stage'е. Решили через **salt'инг** ключа (добавление случайного суффикса к тяжёлым клиентам) и распараллеливание.
2. **Time-aware логика в оконных функциях.** Нужно было гарантировать, что фичи «на дату T» используют **только** данные до T, включая корректную обработку `null`-таймстемпов и сдвинутых snapshot'ов портфеля. Покрыли это юнит-тестами с явными граничными случаями.
3. **Большие справочники vs broadcast.** Часть справочников выросла со временем и перестала влезать в broadcast-лимит → джоба внезапно замедлилась. Добавили мониторинг размеров, автоматический выбор broadcast vs sort-merge join.

## Что бы сделал иначе

- Сразу заложил бы **контракты схемы** с владельцами моделей-потребителей — потом часть фичей пришлось переименовывать/типизировать задним числом.
- Добавил бы **Great Expectations** (или аналогичный data-quality фреймворк) сразу, а не после первого факапа с поехавшими типами в источнике.
- Рассмотрел бы **Delta Lake / Iceberg** вместо чистого Parquet — нативные upsert'ы и time travel сильно упростили бы инкрементальную логику в будущем.
