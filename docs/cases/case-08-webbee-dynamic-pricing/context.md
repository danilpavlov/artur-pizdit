# Контекст

## Webbee ML

**Webbee ML** — аутсорс-компания, которая внедряет ML-решения в бизнес, с фокусом на **e-commerce**. Работа устроена как проектная: под каждого клиента собирается небольшая команда (обычно DS + Backend + PM со стороны клиента), которая проходит цикл «данные → прототип → пилот → продакшн → сопровождение».

## Клиент

**Тип клиента:** селлер, продающий через **Wildberries и Ozon**. Обезличенно: несколько сотен–тысяч SKU в нескольких категориях, своя фабричная/дистрибуционная цепочка, собственная команда категорийных менеджеров, которая до проекта **вручную управляла ценами** через кабинеты маркетплейсов.

## Бизнес-задача

До проекта клиент крутил цены по смеси интуиции и простых правил:
- «конкурент скинул — скидываем и мы»,
- «большой остаток — делаем скидку»,
- «акция WB — идём в промо»,
- «новый товар — держим RRP».

Последствия:
- **Теряли маржу** на товарах, которые спокойно продавались бы и по более высокой цене.
- **Теряли объём** на товарах, где цену держали слишком высоко относительно рынка.
- **Не успевали реагировать** на акции маркетплейсов и действия конкурентов — категорийных менеджеров физически не хватало на сотни SKU.
- **Не было сигналов** о том, что рекомендация «правильная» — решения не проверялись формально.

Задача — собрать систему, которая **ежедневно рекомендует цены по всем SKU**, оптимизируя **маржу** с жёстким ограничением «объём продаж в штуках не должен просесть больше чем на α% относительно baseline». Категорийный менеджер получает список «товар → рекомендуемая цена → ожидаемая маржа → объяснение» и либо принимает автоматически, либо правит вручную.

## Данные

Со стороны клиента были доступны:

- **История продаж по SKU** — из выгрузок кабинетов WB и Ozon, дневные продажи в штуках и деньгах.
- **История цен и акций** — динамика ценника, участие в промо маркетплейсов, применявшиеся скидки и купоны.
- **История остатков на складах** — критично для корректного определения спроса: «нет продаж» ≠ «нет спроса», если товара физически не было.
- **Справочник товаров** — категория, бренд, атрибуты, текстовые описания и изображения для карточек.
- **Себестоимость** — для расчёта маржи `(price − cost) × demand`.

С нашей (Webbee) стороны — **парсер цен конкурентов** с WB/Ozon и внешних магазинов. Сбор делал backend-инженер, продуктовый matching с нашими SKU — отдельная NLP-подсистема на эмбеддингах (см. [флоу работы](flow.md)).

## Почему ML, а не простые правила

- **Правила типа «всегда матчим самого дешёвого конкурента»** ведут к гонке на дно и убивают маржу на товарах, у которых есть узнаваемость бренда / эксклюзивность / лояльная аудитория.
- **Правила по остаткам** не учитывают форму кривой спроса — иногда скидка нужна, иногда нет.
- **Эластичность разная по категориям и даже внутри категории** — её нельзя задать одной формулой, её надо учить из данных.
- **Реакция на акции маркетплейса** сильно зависит от типа товара и бренда — правила не отличают «акция вытянет продажи сама по себе» от «без нашей скидки никто не купит даже в распродажу».

Плюс — клиенту нужна была **автоматизация масштаба**: руками управлять сотнями SKU ежедневно физически нереально.

## Ограничения

- **Официальные API WB и Ozon** — основной канал публикации цен. Есть rate limits, задержки применения цены, иногда модерация. Backend работает с этим, но ML-часть учитывает задержку как часть пайплайна.
- **Обезличенность.** Конкретная категория клиента и название в кейсе не раскрываются (NDA).
- **Бизнес-правила поверх модели** — жёсткие: минимальные цены (MAP/RRP по брендам), запреты на скидку дороже X%, исключения для отдельных SKU (новинки, имиджевые товары). Любая ML-рекомендация проходит через слой правил перед публикацией.
- **Ручной override.** Категорийный менеджер всегда может отклонить или подправить рекомендацию; этот фидбэк собирается и используется для качества.

## Моя роль (Артур)

**Единственный Data Scientist на проекте.** Отвечал за всю ML-часть целиком:

- Forecast спроса (LightGBM global model со всеми SKU).
- Модель ценовой эластичности (по SKU и по кластеру).
- Оптимизация цены с ограничениями.
- NLP-подсистема: кластеризация товаров по описаниям (cold-start и pooled-эластичность) + product matching с конкурентами.
- Бэктест на истории и дизайн A/B-теста по SKU.
- Объяснения рекомендаций для категорийного менеджера.
